# Hospital Data LLM

### AI-Augmented Data Quality Copilot for Healthcare ETL Pipelines

---

## Overview

Hospital Data LLM is an **AI-powered data quality copilot** for healthcare ETL pipelines. It combines a multi-agent LLM architecture with deterministic DuckDB-based validation to detect anomalies, explain data issues in plain language, and support governance-aligned compliance workflows.

Built as a **Design Science Research (DSR) artifact** supporting doctoral research on AI-augmented data quality in healthcare IT.

## Getting Started

### 1. Clone the repository

```bash
git clone https://github.com/kedar-patil-engineer/Hospital_Data_LLM.git
cd Hospital_Data_LLM
```

### 2. Install dependencies

```bash
pip install -r requirements.txt
```

### 3. Configure environment

Create a `.env` file in the project root:

```env
OPENAI_API_KEY=sk-...          # Required for OpenAI provider
API_KEY=your-secret-key        # Required for FastAPI authentication
LIVE_DB_URL=                   # Optional: SQLAlchemy URL for live DB comparison
```

### 4. Load data into DuckDB

Place the 9 CSV files in `data/raw/`, then run:

```bash
python etl/load_duckdb.py
```

This creates `db/hospital.duckdb` with all 9 tables loaded and verified.

### 5. Run the Streamlit app

```bash
python -m streamlit run app.py
```

Opens at `http://localhost:8501`

### 6. Run the FastAPI server (optional)

```bash
uvicorn api:app --reload
```

Opens at `http://localhost:8000` — interactive docs at `/docs`

All endpoints (except `/health`) require the `X-API-Key` header:

```bash
curl -H "X-API-Key: your-secret-key" http://localhost:8000/dq/scan
```

### 7. Run the test suite

```bash
python -m pytest tests/ -v
```

Expected: **90 passed**

## Key Features

**Multi-agent AI pipeline**
Four specialised agents (profiling, DQ rules, SQL generation, narration) collaborate on every query rather than relying on a single monolithic prompt.

**Real confidence scoring**
Every agent response includes a confidence score (0.10–0.98) computed from five runtime signals: result row count, SQL error detection, null density, query complexity, and tool grounding. Not a hardcoded value.

**SQL safety layer**
All SQL is SELECT-only. Write operations, multi-statements, and known bad join patterns are blocked before execution. A schema-aware hallucination guard validates every LLM-generated table and column reference against the real DuckDB schema.

**Healthcare validators**
HL7/FHIR-style checks: FHIR gender enum validation, ICD-10 code pattern matching, CPT code validation, age/DOB mismatch detection, insurance provider sanity checks.

**DQI scoring**
Automated Data Quality Index (0–100) per table: 70% completeness score + 30% anomaly score, with colour-coded visual bars and PDF export.

**Evaluation tracking**
Every agent run is logged to `logs/raw/eval_log.jsonl` with run UUID, provider, model, latency per agent, token counts, and prompt hashes. The evaluation dashboard (pages/eval_dashboard.py) visualises these logs.

**API authentication**
FastAPI endpoints protected by `X-API-Key` header. Development mode auto-detects missing key and warns rather than failing silently.

**Typed error taxonomy**
15 `ErrorType` codes (SQL_BLOCKED, SCHEMA_MISMATCH, LLM_NO_SQL, etc.) replace plain error strings. Machine-readable, log-filterable, governance-aligned.

## Compliance & Governance

- All SQL is **read-only** (SELECT only) — no data mutation possible
- **Schema-aware hallucination guard** prevents LLM-invented table/column references
- Every agent run produces a **traceable log entry** with run UUID and prompt hash
- **Typed error taxonomy** (15 error codes) supports audit and incident response
- API requires **authentication** on all data endpoints
- `.env` with API keys is **gitignored** — never committed
